## 3.3 Medium Severity Findings

### Finding #3: Insecure CORS Configuration (Wildcard Origin)
**Severity:** MEDIUM  
**CVSS v3.1 Score:** 5.3 (AV:N/AC:L/PR:N/UI:R/S:U/C:L/I:N/A:N)  
**Affected Component:** Web Application (`http://192.168.1.49:8082`)  
**CWE:** CWE-693 (Protection Mechanism Failure)  
**OWASP Top 10 (2021):** A05:2021 - Security Misconfiguration  

#### Technical Description
The web server is configured with a wildcard (`*`) for the `Access-Control-Allow-Origin` header. This allows any website to make cross-origin requests to the target application. While this does not directly execute code, it facilitates Cross-Site Request Forgery (CSRF) attacks and allows malicious sites to read sensitive data returned by the API if the application also allows credentials in cross-origin requests.

#### Evidence and Proof of Concept
The following header was observed in the HTTP response:
```http
Access-Control-Allow-Origin: *
```
*Source: `reports/20260314_205043/nmap_20260314_205406_014193.xml`*

#### Impact Analysis
- **Confidentiality:** Low to Moderate. Attackers can read responses if the application allows credentials with the wildcard (which is often blocked by browsers, but misconfigurations exist).
- **Integrity:** Low. Facilitates CSRF attacks where an attacker can force a user's browser to perform actions on the target site.

#### Exploitation Information
- **CISA KEV Status:** N/A
- **CVE Identifiers:** N/A
- **Metasploit Modules:** N/A
- **Exploit-DB References:** N/A
- **GitHub PoCs:** N/A
- **Exploitation Attempt Status:** N/A

#### Remediation Plan
1. **Restrict Origins:** Replace `*` with specific, trusted domain names (e.g., `https://trusted-domain.com`).
2. **Validate Credentials:** If the API requires authentication, ensure `Access-Control-Allow-Credentials` is set to `true` only when specific origins are allowed, never with `*`.
3. **Implement CSRF Tokens:** Ensure all state-changing requests include a CSRF token to mitigate the risk of cross-origin attacks.

### Finding #5: Content Security Policy (CSP) Header Not Set
**Severity:** LOW  
**CVSS v3.1 Score:** 3.7 (AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:L/A:N)  
**Affected Component:** Web Application (`http://192.168.1.49:8082`)

**OWASP Top 10 (2021):** A05:2021 - Security Misconfiguration  

#### Technical Description
The application does not implement a Content Security Policy (CSP) header. CSP is a security standard that helps detect and mitigate certain types of attacks, including Cross-Site Scripting (XSS) and data injection attacks. Without CSP, the browser will execute scripts from any source, making the application more vulnerable to XSS.

#### Impact Analysis
- **Confidentiality:** Low. Increases the likelihood of successful XSS attacks.
- **Integrity:** Low. Allows injection of malicious scripts.

#### Exploitation Information
- **CISA KEV Status:** N/A
- **CVE Identifiers:** N/A
- **Metasploit Modules:** N/A
- **Exploit-DB References:** N/A
- **GitHub PoCs:** N/A
- **Exploitation Attempt Status:** N/A

#### Remediation Plan
1. **Implement CSP:** Add a `Content-Security-Policy` header to all responses.
2. **Strict Policy:** Start with a restrictive policy (e.g., `default-src 'self'`) and relax it only for necessary external resources.
