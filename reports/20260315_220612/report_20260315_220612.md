## 2. Critical Priorities (Must Fix Immediately)
*Focus: Addressing severe code-level vulnerabilities and configuration errors that allow direct exploitation.*

### 2.1 Remediate SQL Injection (SQLi) Vulnerabilities
*   **Context:** 14 instances of SQLi detected (CWE-89) where user input is concatenated directly into SQL strings.
*   **Action Steps:**
    1.  **Refactor Code:** Replace all manual string concatenation for SQL queries with **Prepared Statements** (Parameterized Queries).
        *   *PHP Example:* Switch from `mysql_query("SELECT * FROM users WHERE id='$id'")` to `PDO::prepare("SELECT * FROM users WHERE id=:id")`.
    2.  **Input Validation:** Implement strict allow-list validation for all user inputs before processing.
    3.  **Least Privilege:** Ensure the database user account used by the web application has minimal permissions (no `DROP`, `ALTER`, or `GRANT` rights).
*   **Resources/Tools:** PHP PDO/MySQLi libraries, Code Review.
*   **Estimated Effort:** 4–8 Hours (depending on codebase size).
*   **Security Impact:** **Critical.** Prevents database manipulation, data theft, and potential remote code execution.

### 2.2 Fix Cross-Site Scripting (XSS) Flaws
*   **Context:** 1 instance of XSS detected (CWE-79) where user input is echoed without encoding.
*   **Action Steps:**
    1.  **Output Encoding:** Apply `htmlspecialchars()` or `htmlentities()` to all dynamic data output to the browser.
    2.  **Context-Aware Encoding:** Ensure encoding matches the context (HTML body, attribute, JavaScript, URL).
    3.  **Content Security Policy (CSP):** Implement a strict CSP header to restrict script execution sources (see Medium-term section).
*   **Resources/Tools:** PHP `htmlspecialchars`, CSP generator tools.
*   **Estimated Effort:** 2–4 Hours.
*   **Security Impact:** **High.** Prevents session hijacking, defacement, and malware injection.

### 2.3 Secure Web Server Configuration (Information Disclosure)
*   **Context:** Server banners reveal specific versions of Apache, Nginx, and PHP (e.g., `Apache/2.4.66`, `PHP/8.5.4`), aiding attackers in finding version-specific exploits.
*   **Action Steps:**
    1.  **Apache:** Edit `httpd.conf` or `apache2.conf`:
        *   Set `ServerTokens Prod`
        *   Set `ServerSignature Off`
    2.  **Nginx:** Edit `nginx.conf`:
        *   Set `server_tokens off;`
    3.  **PHP:** Edit `php.ini`:
        *   Set `expose_php = Off`
    4.  **Restart Services:** Restart Apache and Nginx to apply changes.
*   **Resources/Tools:** Web server configuration files.
*   **Estimated Effort:** 1 Hour.
*   **Security Impact:** **High.** Reduces the effectiveness of reconnaissance and targeted attacks.

## 3. Medium-term Improvements
*Focus: Strengthening the application's defense-in-depth and fixing configuration gaps.*

### 3.1 Implement Security Headers
*   **Context:** Missing CSP, Anti-Clickjacking, and MIME-Sniffing headers.
*   **Action Steps:**
    1.  **Content Security Policy (CSP):** Configure a strict CSP header (e.g., `default-src 'self'; script-src 'self';`).
    2.  **X-Frame-Options:** Set to `DENY` or `SAMEORIGIN` to prevent clickjacking.
    3.  **X-Content-Type-Options:** Set to `nosniff` to prevent MIME type sniffing.
    4.  **X-XSS-Protection:** Set to `1; mode=block` (legacy support).
*   **Resources/Tools:** Web server config, `.htaccess`, or application framework middleware.
*   **Estimated Effort:** 2–3 Hours.
*   **Security Impact:** **Medium.** Mitigates XSS and Clickjacking attacks even if code vulnerabilities persist.

### 3.2 Secure Cookie Configuration
*   **Context:** Cookies lack `SameSite` and `Secure` attributes.
*   **Action Steps:**
    1.  **Enable HTTPS:** Ensure the application is served over HTTPS (required for the `Secure` flag).
    2.  **Update PHP Session Config:** In `php.ini` or `session_start()`:
        *   `session.cookie_httponly = 1`
        *   `session.cookie_secure = 1`
        *   `session.cookie_samesite = Strict`
*   **Resources/Tools:** PHP configuration, SSL/TLS certificates.
*   **Estimated Effort:** 2 Hours.
*   **Security Impact:** **Medium.** Protects session tokens from CSRF and interception.

### 3.3 Remove Sensitive Files from Web Root
*   **Context:** `php.ini` and `robots.txt` are accessible, potentially revealing configuration details.
*   **Action Steps:**
    1.  Move `php.ini` and other configuration files outside the web-accessible document root.
    2.  Configure the web server to deny access to files starting with a dot (e.g., `.git`, `.env`) or specific extensions like `.ini` if they must remain in the root.
    3.  Review `robots.txt` to ensure it does not expose sensitive paths (e.g., `/admin`, `/backup`).
*   **Resources/Tools:** File system, Web server config.
*   **Estimated Effort:** 1 Hour.
*   **Security Impact:** **Medium.** Prevents accidental disclosure of system configuration.
