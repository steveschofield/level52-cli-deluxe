#### 3.1 Critical Findings

Finding ID: CRIT-001 Title: Critical Exposure of Sensitive Backup and Source Files Severity: CRITICAL Affected Component: Web Server Root (`/`) CVSS v3.1: 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H) OWASP Top 10: A01:2021 - Broken Access Control CWE: CWE-530 (Information Disclosure of Sensitive Data)

Technical Description: The web server is misconfigured to allow direct access to sensitive backup files and source code artifacts. Automated scanning identified the presence of a SQL database dump (`/database.sql`) and a private certificate file (`/site.pem`). These files are publicly accessible without authentication, exposing the entire database schema, user credentials, and potentially private cryptographic keys.

Privilege Escalation: Access to private keys (`site.pem`) could allow an attacker to impersonate the server or decrypt traffic. Source Code Disclosure: Exposure of application logic, internal IP addresses, and hardcoded secrets.

Exploitation Information: CISA KEV Status: N/A CVE Identifiers: N/A Metasploit Modules: N/A Exploit-DB IDs: N/A GitHub PoCs: N/A Exploitation Attempt: Not attempted (Passive discovery of accessible files).

Remediation Plan: 1. Immediate Action: Remove or restrict access to `/database.sql` and `/site.pem` immediately. 2. Configuration: Configure the web server (Nginx/Apache) to deny access to files with extensions `.sql`, `.pem`, `.bak`, `.old`, and `.git`. 3. Audit: Scan the entire directory structure for other sensitive files (e.g., `.env`, `.htaccess`, `config.php`).

Finding ID: CRIT-002 Title: Authentication Bypass on Protected API Endpoint Severity: CRITICAL Affected Component: API Endpoint (`POST /orders`) CVSS v3.1: 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H) OWASP Top 10: A01:2021 - Broken Access Control CWE: CWE-287 (Improper Authentication)

Technical Description: The API endpoint `POST /orders` is documented in the OpenAPI specification as requiring authentication. However, the implementation fails to enforce this requirement. The server returns a `200 OK` status code even when the `Authorization` header is missing or contains invalid credentials, effectively bypassing the authentication mechanism.

Impact Analysis: Unauthorized Access: Any unauthenticated user can create orders, modify data, or access restricted resources. Data Integrity: Attackers can inject malicious data into the order system. Compliance Failure: Violates security requirements for authenticated access to sensitive operations.

Exploitation Information: CISA KEV Status: N/A CVE Identifiers: N/A Metasploit Modules: N/A Exploit-DB IDs: N/A GitHub PoCs: N/A Exploitation Attempt: Successful (Automated tool confirmed bypass).

Remediation Plan: 1. Code Review: Audit the middleware handling the `/orders` endpoint to ensure the `Authorization` header is strictly validated. 2. Enforcement: Implement a "fail-secure" default where requests without valid tokens return `401 Unauthorized`. 3. Testing: Re-run the Schemathesis scan to verify the fix. 4. Logging: Ensure all failed authentication attempts are logged for monitoring.
