#### 3.1. Critical: Sensitive File Exposure (Backups, Keys, and Dumps)
**Severity:** CRITICAL  
**Affected Component:** Web Server Root (`/`)  
**CVSS v3.1:** 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H)  
**OWASP Top 10:** A01:2021 - Broken Access Control  
**CWE:** CWE-530 (Use of Hard-coded Credentials / Sensitive Data Exposure)

**Technical Description:**  
The web server is misconfigured to allow direct access to sensitive files that should be restricted. Automated scanning identified the presence of database backups and private key files in the public web directory. Specifically, `/backup.sql` and `/database.pem` are accessible without authentication. Exposure of these files can lead to full database compromise, credential theft, and unauthorized access to backend systems.

**Evidence & Proof of Concept:**  
The `nikto` scanner successfully retrieved the following files:
*   `/backup.sql`: Potentially interesting backup/cert file found.
*   `/database.pem`: Potentially interesting backup/cert file found.

**Impact Analysis:**  
*   **Confidentiality:** High. Attackers can download the entire database schema and data.
*   **Integrity:** High. Private keys (`.pem`) allow attackers to impersonate the server or decrypt traffic.
*   **Availability:** High. Database dumps can be used to reconstruct the environment or facilitate ransomware attacks.

**Exploitation Information:**
*   **CISA KEV Status:** Not listed as actively exploited in the wild (CISA KEV) in current data.
*   **CVE Identifiers:** N/A
*   **Metasploit Modules:** N/A
*   **Exploit-DB IDs:** N/A
*   **GitHub PoCs:** N/A
*   **Exploitation Attempt:** Successful. The scanner confirmed the files are publicly accessible via HTTP GET requests.

**Remediation Plan:**
1.  **Immediate Action:** Remove or rename the exposed files (`/backup.sql`, `/database.pem`) from the web-accessible directory immediately.
2.  **Configuration:** Configure the web server (Nginx/Apache) to deny access to files with extensions `.sql`, `.pem`, `.bak`, `.dump`, and `.log`.
3.  **Access Control:** Ensure sensitive files are stored outside the document root or in a directory with strict `Allow`/`Deny` directives.
4.  **Verification:** Re-scan the target to confirm the files return a `403 Forbidden` or `404 Not Found` status.

#### 3.2. High: Broken Access Control - Authentication Bypass on Order Creation
**Severity:** HIGH  
**Affected Component:** API Endpoint `POST /orders`  
**CVSS v3.1:** 7.5 (AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:H/A:N)  
**OWASP Top 10:** A01:2021 - Broken Access Control  
**CWE:** CWE-287 (Improper Authentication)

**Technical Description:**  
The API endpoint responsible for creating orders (`POST /orders`) fails to enforce authentication requirements. The server accepts requests without a valid `Authorization` header or with invalid credentials, returning a `200 OK` status instead of the expected `401 Unauthorized`. This allows unauthenticated actors to create orders, potentially leading to inventory manipulation, financial fraud, or data injection.

**Evidence & Proof of Concept:**  
*   **Test Case 1:** Request sent with invalid authentication token.
    *   *Expected:* `401 Unauthorized`
    *   *Actual:* `200 OK`
*   **Test Case 2:** Request sent with missing `Authorization` header.
    *   *Expected:* `401 Unauthorized`
    *   *Actual:* `200 OK`

**Impact Analysis:**  
*   **Integrity:** High. Attackers can create fraudulent orders or manipulate business logic without authorization.
*   **Confidentiality:** Low (in this specific context), but the bypass may extend to other endpoints.
*   **Business Impact:** Potential for financial loss, inventory depletion, and data integrity violations.

**Exploitation Information:**
*   **CISA KEV Status:** Not listed.
*   **CVE Identifiers:** N/A
*   **Metasploit Modules:** N/A
*   **Exploit-DB IDs:** N/A
*   **GitHub PoCs:** N/A
*   **Exploitation Attempt:** Successful. The `schemathesis` tool confirmed the API accepts unauthenticated requests.

**Remediation Plan:**
1.  **Middleware Implementation:** Enforce strict authentication middleware on the `POST /orders` endpoint.
2.  **Validation:** Ensure the server validates the `Authorization` header and rejects requests with missing or invalid tokens with a `401` status code.
3.  **Schema Enforcement:** Update API schema definitions (OpenAPI/Swagger) to explicitly require authentication for protected endpoints.
4.  **Testing:** Perform regression testing to ensure all protected endpoints require valid credentials.

#### 3.5. Medium: Overly Permissive Cross-Origin Resource Sharing (CORS)
**Severity:** MEDIUM  
**Affected Component:** Web Server / API (`http://192.168.1.49:3000`)  
**CVSS v3.1:** 4.3 - 5.4 (AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N)  
**OWASP Top 10:** A05:2021 - Security Misconfiguration  
**CWE:** CWE-942, CWE-693

**Technical Description:**  
The server responds to requests with the HTTP header `Access-Control-Allow-Origin: *`. This wildcard configuration allows any website on the internet to make cross-origin requests to this server and read the responses. While not a direct exploit, this facilitates attacks such as Cross-Site Request Forgery (CSRF) or data exfiltration if the application relies on browser-based authentication (cookies) and does not implement proper CSRF tokens.

**Evidence & Proof of Concept:**  
*   **Tool:** `nmap`, `headers`, `idor-scanner`
*   **Header Found:** `Access-Control-Allow-Origin: *`
*   **Reasoning:** The server explicitly allows any origin to access resources.

**Impact Analysis:**  
*   **Confidentiality:** Low/Medium. Malicious sites can read API responses if the user is logged in.
*   **Integrity:** Low. Potential for CSRF if tokens are missing.
*   **Availability:** Low.

**Exploitation Information:**
*   **CISA KEV Status:** Not listed.
*   **CVE Identifiers:** N/A
*   **Metasploit Modules:** N/A
*   **Exploit-DB IDs:** N/A
*   **GitHub PoCs:** N/A
*   **Exploitation Attempt:** Not attempted; configuration confirmed via header inspection.

**Remediation Plan:**
1.  **Restrict Origins:** Replace `*` with specific, trusted domain names (e.g., `Access-Control-Allow-Origin: https://trusted-domain.com`).
2.  **Credentials:** If credentials are required, ensure `Access-Control-Allow-Credentials` is set to `true` and `Access-Control-Allow-Origin` is not `*`.
3.  **CSRF Protection:** Implement CSRF tokens for state-changing operations to mitigate the risk of cross-origin requests.
